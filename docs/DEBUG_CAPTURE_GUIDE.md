# 华为耳机通用协议采集指南

HuaweiPods 目前只有 HUAWEI FreeBuds 3 的控制协议经过验证。FreeBuds、FreeClip、FreeLace 等华为耳机即使同属一个品牌，服务 UUID、命令编号、能力位、参数范围和返回格式也可能因系列、代际和固件而不同，因此不能仅凭设备名称复用 FreeBuds 3 的写指令。

项目提供两个构建：

- `release`：正式版，不注入智慧生活/智慧音频，也不包含采集界面和接收器。
- `debug`：适配采集版，在用户明确启用 LSPosed 作用域后，仅观察官方 App 调用 Android 蓝牙 API 时产生的 GATT、RFCOMM 和 L2CAP 数据，不修改、阻断或重放官方 App 的调用。

Debug 与 Release 使用相同应用 ID，不能同时安装，这可以避免两个 Xposed 模块重复 Hook。同一签名的两个 APK 可以直接互相覆盖；出现“签名不一致”时，必须改用维护者提供的同签名 APK，或卸载原版本后再安装。

## 开始前

1. 安装维护者提供的 `HuaweiPods-*-capture-debug.apk`。
2. 在 LSPosed 中启用 HuaweiPods，并至少勾选：
   - 平时管理耳机的 `智慧生活`（`com.huawei.smarthome`）或 `智慧音频`（`com.huawei.smartaudio`）；
   - HuaweiPods 原有的系统作用域。
3. 强行停止所选官方 App 后重启手机，确保新作用域生效。
4. 断开手表、车机和其他蓝牙设备，只保留待适配耳机；采集期间不要接打电话、录音或播放涉及隐私的音频。
5. 先在系统蓝牙中连接待适配耳机，再打开桌面上的“HuaweiPods 协议采集”。授予“附近的设备”权限后，向导会读取当前已连接的蓝牙音频设备名称：只有一个时自动填入，有多个时由测试者选择；读取失败时可以手动填写系统蓝牙中显示的名称。
6. GitHub issue 为可选项。已有适配 issue 时可填写编号或完整链接，没有时可以留空并在导出后新建 issue。

自动填入名称只用于标记采集对象，不代表向导能够识别具体型号或判断设备支持哪些能力。请以官方 App 当前实际显示的功能为准。

QQ群 `1022359908` 只用于待适配型号登记、测试招募、操作答疑和传输协调。群聊属于公共场所，禁止直接发送未经检查的采集 ZIP、HCI 日志或完整 bugreport。

## 引导采集

先完成页面中的四项准备确认，再点“开始本次采集”。每一个功能都严格按以下节奏操作：

1. 在采集向导中点“开始此项并打开官方 App”。
2. 在官方 App 中只改变卡片要求的一个设置。
3. 等待至少 2 秒，让耳机回包。
4. 返回采集向导，点“完成并标记”。

通用向导列出了华为耳机系列可能出现的功能全集，但不会自动判断当前设备的具体能力。官方 App 没有显示或设备不支持某项功能时，必须点“无此功能，跳过”，不要为了凑齐步骤反复尝试。这样日志能区分“未操作”和“设备不支持”。

向导可能包含连接与设备信息、电量、声音控制、手势、佩戴检测、自适应音频、头动/语音控制、均衡器、空间音频、音质、低时延、双设备、充电盒、耳塞贴合和 FreeLace 磁吸设置等项目。每一项都只操作官方 App 当前可见的一个设置，不显示就跳过。

以下操作无论页面是否出现都不属于本采集流程，禁止执行：

- 升级固件、恢复出厂设置、重置耳机或取消配对；
- 发起/接听真实通话、录音、发送语音或采集听力/健康数据；
- 打开或记录华为账号、手机号、健康、位置、通知内容等隐私页面；
- 使用“查找耳机”或任何会让耳机响铃、上传位置的功能。

完成后点“停止采集”，先使用默认选项导出 ZIP。压缩包包含：

- `metadata.json`：手机、Android、HuaweiPods 版本、设备名称和可选 issue；
- `events.jsonl`：按时间排序的操作标记和蓝牙事件；
- `README.txt`：文件说明和隐私提醒。

蓝牙地址会默认脱敏，但原始协议载荷仍可能包含设备标识。提交前务必解压检查；不要上传账号、健康、位置、通知内容页面、完整系统日志或与目标耳机无关的数据。

停止页会单独显示“协议事件”数量。该数字为 `0` 时，ZIP 里只有向导 marker，不能用于推导协议，应按下一节排查作用域或使用 HCI 兜底。

## 提交与反馈

1. 先搜索 [HuaweiPods Issues](https://github.com/Nshpiter/HuaweiPods/issues) 是否已有同型号记录。
2. 已有 Issue 时，在采集向导填写编号并回到原 Issue 跟进；没有时可留空，导出、检查后再新建 Issue。
3. Issue 中请提供耳机在系统蓝牙里显示的名称、手机型号、系统版本、官方 App 及版本、`protocol_event_count`、每项操作的实际表现，以及必要的脱敏采集包。
4. 普通采集 ZIP 只有在解压检查并确认可公开后，才可附到 GitHub Issue；拿不准时先只提交环境信息，等待维护者安排私下传输。
5. `protocol_event_count` 为 `0` 时只报告环境和排查结果，不要把仅含 marker 的 ZIP 当作有效协议样本。

HCI 日志和完整 bugreport 可能包含同一时段内其他设备、账号或通信数据，无论是否初步检查，都禁止上传到公开 Issue 或 QQ 群。仅在维护者明确要求后，通过双方确认的私下渠道传输。

## 结束测试环境

1. 确认采集包已经导出并妥善保存。
2. 在 LSPosed 中取消智慧生活 / 智慧音频的 HuaweiPods 作用域，重启对应进程；不确定是否生效时直接重启手机。
3. 换回维护者提供的同签名 Release 版，或卸载 Debug 版。Debug 与 Release 使用相同应用 ID，不能同时安装。
4. 若附加过 HCI 日志或不再需要本地采集记录，请清除 Debug 版应用数据，删除应用私有目录中的副本。

## 没有记录到协议事件时

先确认 `events.jsonl` 中除了 `session_start`、`marker`、`session_stop` 外是否存在 `protocol_event`。如果完全没有：

1. 在 issue 中说明手机型号、系统版本、官方 App 版本和“只有 marker，没有 protocol_event”。
2. 确认 LSPosed 确实勾选了正确的官方 App，并重启后再做一次最短采集。
3. 仍为空时，说明该版本可能走了厂商服务或 native 通道。这时才使用蓝牙 HCI snoop 兜底。

Android 开发者选项提供“启用蓝牙 HCI 信息收集日志”。启用前断开其他蓝牙设备，只重复一个明确操作，完成后立即关闭。Debug 向导默认不会读取 HCI；只有维护者明确要求并且用户勾选高风险确认时，才会尝试附加现有日志，且不会自动修改开发者选项。

HCI 日志可能包含同一时段内所有蓝牙设备的地址、名称和通信内容。不要公开完整 bugreport，也不要发到 QQ 群或 GitHub Issue；先检查和脱敏，再按维护者明确指定的私下渠道传输。勾选附加后，副本会暂存在 HuaweiPods 的应用私有目录；确认不再需要时可清除应用数据或卸载 Debug 版。Android 官方说明见 [Configure on-device developer options](https://developer.android.com/studio/debug/dev-options#debugging)。

## 维护者分析顺序

1. 用 `operation.*.start` 与 `operation.*.done` marker 划出每次单变量操作的时间窗。
2. 比较同一功能“关闭 → 开启”或不同档位之间新增/变化的 TX 帧。
3. 用随后的 RX 帧确认应答、状态通知和校验字段。
4. 至少用第二轮采集复核稳定字段与计数器/时间戳字段。
5. 只把已在真机验证的命令加入对应型号路由；未知型号继续保持只读和不可写。
